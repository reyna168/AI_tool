現在有85%的專業開發者在用 AI Coding Agent, 41%的新Code是 AI寫的,
但 Vibe Coding, Agents and Engineering這些詞, 十個人講有十種意思,有人說 Vibe Coding是未來,有人說那是垃圾Code的製造機。
Google最近上線了一套五天的 AI 開發課程, 第一次把整個業界正在收斂的共識寫成一套正式的框架,光是第一天的講義就有51頁。
今天我會用15分鐘把 Day 1 的核心整理給你。 這支影片分成四個部分,
第一 Vibe Coding 跟 Agents and Engineering 的差別, 第二比 Prompt Engineering 更重要的技能, Contacts Engineering,
第三 Agent 等於 Model 加 Harness 以及為什麼 Agent 出包時大多數狀況都不是因為選錯模型,
第四 Token 經濟學,為什麼在早期就開始投資工作系統的人, 後期使用 AI 的成本會更便宜?
看完這支影片,你會得到一套完整的 AI 開發新字模型,那我們廢話不多說,直接開始。
先從 Vibe Coding 這個詞講起,
2025年2月, Karpati 發了一則貼文,描述一種全新的寫成式方式,就是完全順的感覺走,
跟 AI 用自然語言描述你要什麼,完全不看任何的 Code,如果遇到錯誤,就把錯誤訊息直接複製給 AI 自己修。
這個詞會爆紅,是因為它描述了很多人早就開始使用的 AI 開發方式,只是一直沒有一個詞可以描述它,
但爆紅的代價就是被濫用,一個資深工程師用 AI 實作一個規格明確的功能算 Vibe Coding 嗎?
一個團隊用 Agent 執行規劃好的架構,算嗎?這個詞被套到什麼都能講,反而什麼都說不清楚。
所以到了 2026 年初, Karpati 自己又補了一個詞叫做 Agentic Engineering 用來描述有紀律的那一端。
有看過我之前影片的朋友,這個詞你應該不陌生。
Google 這門課程的第一個核心主張,就是這兩個東西不是二選一的開關,是一條光譜。
光譜上有三個位置, Vibe Coding, Structure AI Assisted Coding, Agentic Engineering,
而且判斷標準不是你用不用 AI,是 AI 的輸出周圍有多少的結構、驗證以及人類判斷。
講義理給了一張對照表,第一個是 Intent 的規格化程度。
Vibe Coding 是隨口的自然語言, Prompt Agentic Engineering 是正式的Spec 架構文件 Memory Files。
第二個是驗證方式, Vibe Coding 的驗證是看起來會動,
Agentic Engineering 是自動化測試, CICD Gates 加上 LM Judges。
第三個是錯誤處理, Vibe Coding 是錯誤訊息,貼回去叫 AI 修,
Agentic Engineering 是 Agent 在你定義好的邊界內自我診斷,而我們人只處理架構成績的問題。
至於你該站在光譜的哪個位置,其實沒有對錯,單純看你的使用場景以及出錯的風險。
周末做個 Prototype,純 Vibe Coding 完全合理,跑壞就重來,沒人受傷。
但一個處理金流的 Production API 就必須用 Agentic Engineering。
你跟 CTO 說我們在 Vibe Coding 付款系統,他可能臉都濾了。
但你說我們在做 Agentic Engineering AI 負責實作,人類設計約束,測試覆蓋確保證確性,這聽起來就沒那麼膽戰心驚。
整條光譜上,兩端最大的分水嶺是驗證,而驗證有兩種。
Tests 驗證確定性的部分,這個 Function 給這個輸入就該 2 這個輸出。
Evils 驗證非確定性的部分,Agent 走的路徑對不對,工具選的對不對,最後的產出有沒有到品質標準。
Google 講的很死,沒有這兩個東西,不管你的 Prompt 寫得多精緻,你做的都還是 Vibe Coding。
那如果你想往 Agentic Engineering 大段移動,概念什麼呢?
答案不是把 Prompt 寫得更漂亮,而是學會 Context Engineering。
Context Engineering 可以想成是幫新人工作入紙簡報,一個新人來報到。
你不會只丟一句幫我把這個功能做出來,你會告訴他任務是什麼,專案背景是什麼,公司有哪些規範。
AI 也是一樣,他寫出來的 Code 的品質,跟你給他哪些 Context 的關聯性很大。
所以最關鍵的點在於,一個新加入團隊的工程師需要知道什麼才能有效貢獻,而我又要怎麼把這些知識變成 AI 能用的形式。
Context 一共分成六種, Instructions 負責定義 Agent 的角色和邊界, Knowledge 是給 Agent 的領域知識, Memory 是短期和長期的狀態, Example 是行為示範,
Tours 是他能呼叫的工具,定義, Godrails 是硬性約束。
而這六種 Context 又可以很漂亮地分成 Static Context 跟 Dynamic Context。
Static Context 是每次都一定會載入的東西,系統指令, Rule Files, 像 Agents Markdown, Club Markdown 這種檔案。
好處是可靠 Agent 不用自己去找,壞處是貴,因為不管什麼問題,必須載入,而載入就會消耗 token。
Dynamic Context 是按需求載入的 Context,像是 Skills, Rack, 撈回來的文件,工具, 執行的結果。
Dynamic Context 的好處是便宜,可擴展,需要才付錢,風險是 Agent 該去抓的時候沒去抓。
至於哪些 Context 放 Static,哪些放 Dynamic,這條邊界本身就是一個關鍵的架構決策,要像 Code 一樣,被 Review, 被板控。
而管理 Dynamic Context 最強的 Pattern 就是 Agent Skills。
與其把所有專業知識塞進 System Prompt,不如讓 Agent 平常保持通用型,需要的時候再讀取 Skill,變成特定任務的專家。
這個機制叫 Progressive Disclosure。
Agent 啟動時只看到每個 Skill 的一行 metadata,任務匹配到了才再入完整指令,需要深層參考資料才去拉。
結果就是一個 Agent 可以帶著幾十種專業能力,但只為正在用的那一個副 token 成本。
關於 Skill 我之前出過滿多影片講解相關的製作技巧,但這邊還是快速補充兩點。
第一, Skills 有負利效應。
你做一個每天都會用到的 Skill,用的過程中發現產出不如預期,就回頭改它。
一個月之後,這份 Skill 已經比一開始好用非常多,所以不要想著一步到位,要持續疊帶。
第二, Skills 要寫的 Agent 有善,人類可維護。
因為一個任務跑下來, Agent 可能會調用好幾份 Skills。
當產出走歪的時候,你要有能力找出是哪一份撈出時 Skill 把 Agent 帶歪了,
所以不要寫一份一萬行的 Skill,你連看都看不完,更不用說維護了。
Contest Engineering 就是 Vibe Coding 通往 Agent Engineering 的橋。
而 Google 這門課程在 Day 3 的時候有一整篇專門講 Session, Skills, Memory,那一集我們在挖身。
講完這些後,我們把視角拉高,看整個開發流程會變成什麼樣子。
SDLC 這個詞如果你不是技術背景可能沒聽過,全文是 Software Development Life Cycle
就是從需求、設計、實作、測試、部署到維護的完整軟體開發流程。
這條流程因為 AI 的出現被不斷壓縮,但壓得很不均勻。
Implementation 也就是實際寫 Code 的部分,從幾週變成幾小時,
但需求訪談、架構決策、驗證品質,這些大多數還是人的速度。
所以不是就流程被加速了,而是誕生了一個新的流程。
新流程中的每個階段之間的邊界變模糊,
疊帶週期從週變成分鐘,而 SPEC 的品質變成新的瓶頸。
需求階段以前是文件在部門之間傳來傳去,現在變成人跟 AI 的對話,
訪談還是要人親自談,但一談完 AI 幾分鐘就能把 SPEC 跟出版的 Prototype 伸出來。
架構階段這是最頑固的人類階段,因為架構決策本質上是 trade-off,
一致性還是可用性,自己做還是買現成的,這些取捨依賴商業脈絡, AI 抓不到全貌。
AI 向常的是架構定案之後的執行。
實作階段的數據業界調查說,生產力提升25%到 39%,
但 Metal 有一個研究發現,資深工程師用 AI 做某些任務,反而慢了 19%,
因為時間都花在驗證和修正 AI 的產出。
這兩個數據不衝突,他們同時說明了一件事, AI 不是消滅實作工作,
是把實作重寫,變成 review,引導,驗證,還有一個最被低估的階段,維護。
以前那種只有原作者看得懂,沒人感動的 legacy code,
現在 agent 可以獨動整個 Cobase,理解它的 pattern,在尊重既有架構的前提下動手改。
框架遷移,更新過時的 API,現代化測試,這些以前風險太高,
根本沒人想碰的是,現在可以開始著手翻修了。
把這些變化串起來,Google 給了一個新字模型,叫 Factory Model 工廠模型,
意思是把整個開發流程想成做工廠,而你是工廠經理。
工廠經理不會親手組裝每一個零件,它負責設計產線,把官品質,產品是產線自己跑出來的。
放到軟體開發上,開發者的主要產出就不再是程式碼,而是產出程式碼的系統。
這個系統包含 spec 跟 context,負責實做的 agents,驗證證確性的測試跟品質關卡,
把失敗導回去修正的 feedback loops,還有約束行為的 guardrails。
你給 agent 是 success criteria,不是 step by step 指令,然後讓它自己疊帶。
那這座工廠裡的核心機器,也就是實際在產線上幹活的 agent 本體,到底長什麼樣?
Google 給出的答案是 harness.
很多人把model 當成系統本身,新model 出來,就覺得 agent 變聰明了,用舊model 就覺得 agent 變笨了,model 變成一切好壞的解釋。
這個觀念是錯的,而且它會讓你把時間投資在錯的地方。
正確的公式很簡單, agent 等於model 加 harness.
一個 role model 不是 agent,它要有 harness 給它狀態,給它資金工具的能力,給它 feedback loop 給它可執行的約束,它才變成一個 agent.
你平常用 global cursor codex 感受到的行為差異,很大一部分是 harness 決定的,不只是底下那顆model。
還記得前面說 context engineering 是新員工的入職簡報嗎?
Harness engineering 就是整建公司的運作方式,IT 基礎設施,工作流程規範,門禁,績效評估系統,全部算在內,入職簡報只是其中一環。
具體來說 harness 有六大鍵。
第一, rule files,定義 agent 是誰,在乎什麼,什麼事情絕對不能做。
第二, tools 它能呼叫的 function, mcp servers 還有告訴它什麼時候該用哪個的說明。
第三, sandbox 它的 code 在哪裡跑,能摸到什麼,摸不到什麼。
第四, orchestration sub agent 的調度,model 之間的路由,專家之間的交接規則。
第五, hooks 在生命週期的固定點跑的確定性code,比方說 commit 之前自動擋掉硬編碼的密碼, hooks 放的是那些 agent 不該忘卻常常忘的是。
第六, observability, logs, traces, evals, 成本監控。
沒有這一層,你根本不知道 agent 是做得好,還是在偷偷浪費你的錢亂做一通。
Google 的課程中給了我們兩個 case 證明 harness 的重要性。
Terminal Bench 2.0 是一個很硬的 coding agent benchmark,有一個團隊完全不換model,只改 harness,就把成績從 30 名以外拉進前五名。
另一個是 lanechain 的實驗,同一顆model,只調整 system prompt, tools 跟 middleware,加了 13.7分。
同一顆nau,換一套工作環境,表現天差地遠。
大部分的 agent 失敗都是因為 configuration, agent 出包的時候你的第一反應是怪model,打開排行榜想換一顆,但真正的原因通常是缺一個工具,
一條規則寫得太模糊,少一個 guardrail,或者 context 塞滿了雜訊。
同一個 agent 你可以拿它 vibe coding,也可以拿它做 agentic engineering,
差別就在你花多少時間建置自己或者團隊的 harness。
如果放到日常的習慣, agent 出包的時候不要修完 bug 就走,多花五分鐘,回頭問自己一句,我的 rules, workflows, skills 哪裡可以改,
讓這種錯誤不再發生,把答案寫回你的 harness。
這樣沒考一輪,你的系統就更可靠一點,錯誤也可以從成本變成資產。
最後一個重點,這一大片 harness 是你跟團隊的地盤,不是model 廠商的,model 你控制不了, harness 是你唯一能控制,也最值得投資的地方。
當你把這些東西都做到好,系統有能力自己運作,那人的角色是什麼呢?
Google 給出的答案是conductor 跟 orchestrator 他們認為人的角色會在這兩者之間來回切換。
conductor 是指揮家模式,你在 IDE 裡看著 code 一行一行出現,隨時下指令,隨時修正,每一步都在你的掌控裡。
這個模式適合複雜邏輯,極守的 debug 還有你不熟的 codebase,因為這些情境你需要理解每一個改動。
Orchestrator 則是更放牛吃草,你定義目標後只派任務給 agents,他們在背景平行跑,可能同時處理 codebase 的不同部分,你隔一段時間回來 review 結果,給方向修正。
這個模式適合定義名區的任務,bugface,照著既有 pattern 做的功能,codebase,遷移,測試生成。
你會在這兩個模式之間流動切換,而 orchestrator 模式需要的技能有四個,第一個是 specification,把任務定義到 agent 不會誤解。
第二個是 decomposition,把大任務拆成 agent 在一個 session 中能夠消化的大小。
第三個是 evaluation,快速判斷產出過不過關。
最後一個是 system design,設計約束,測試,feedback loop,讓 agents 保持高產。
道理都懂了之後,回歸到建置 harness,寫 evals 的時間成本,真的值得嗎?
Google 這道課程從 token economics 的角度來回答這個問題,用兩個財務概念, capex 跟 apex,前期投資和營運成本。
Vibe coding 看起來超便宜,一個月的訂閱費,幾句 prompt 就能開工,前期投資趨勁於零,但它藏著三個會負力成長的營運成本。
第一個是 token 燃燒率,沒有整理過的 context,整報到進去,然後反覆叫 model show 它自己沒被驗證過的錯,這個低成功率的回圈每一輪都在燒 API 費用。
第二個是維護稅,沒有結構一致性的 AI code,半年後出 bug,工程師要花好幾天逆向工程納陀意大利面。
第三個是資安補救,code 升得快,漏洞也會多,production 環境,修一個資安漏洞的成本,是設計階段抓到了好幾倍。
Agentic Engineering 把這套帳整個反過來。
前期要投工程時間,設計 API schema,建設式套件,整理 context, capex 高,但每個功能的編輯成本大幅下降,因為 AI 是在一座治理好的工廠裡跑。
坦出天生結構就是對的,預先測過的,符合公司標準的。
Contact Engineering 不只是技術,它是財務槓桿。
LM 是按你送進去的每一個 token 收費的。
把 10 萬 token 的 report 整包塞進每一個 prompt,從 token 效率來說是不友善的,而一份精準的文件、提示詞,或者任何 context 會直接拉高 First Pass 成功率。
第一次就做對等於省掉整條 trial and error 的錢。
你不能決定一個模型的費用,但你可以用比較少的 token 書完成一樣的任務,只要你管理好 context。
影片的最後,我想總結一下 Google 在 Day 1 課程中給出的行動建議,讓你看完這部影片後能夠有所行動。
如果你是個人開發者,第一,可以開始建立並維護你自己的 Agents Markdown 或 Cloud Markdown,實行就可以開始,技術戰、慣例、硬規則、workflow。
然後 Agents 每做一次你不想再看到的事,就加一條規則。
第二,測試跟 Evol 實在生 code 之前寫,他們是你跟 AI 之間的合約,一份好的測試套件,比任何自然語言的 prompt 都更能精確的傳達你的意圖。
第三,要上線的 code,每一行都要 review。對看起來很聰明的東西保持懷疑,檢查 import 的套件是不是真的存在。
第四,基本功不能忘記, debug 的方法、系統設計的原則都要留著,因為 AI 是放大你的這些專業,不是拿來替代它。
而如果你是 vibe code 的半路轉行,這些東西是你需要花時間去了解的。
如果你是帶團隊的主管,或者在幫公司規劃 AI 轉型,第一件事情是把 AI 開發當成工程投資,不是生產力功能,導入 coding agent,卻不配套 evols, observability 跟架構標準,只會產出有速度沒品質的 code。
技術在堆的比誰都快。
第二,把 harness 當成團隊的共用資產, system prompt, skill pool, evol 套件都要像 code 一樣被板控、被 review 有人負責維護,這些東西建議此,之後每個專案都在負利。
第三,人跟 agent 的混合團隊會是常態,我們人類定出方向, agent 則是負責實作。
另外,招募和培養人才的重心也會從實作能力移到判斷力,會寫最多 code 的工程師,不再是最有價值的工程師。
能把 agent 指揮的好的才是,課程講義中的最後一句話,我認為說得很好,在這邊分享給各位, generation is solved, verification, judgment, and direction are the new craft,
城市馬產出的效率問題已經被解決了,而驗證、判斷、方向才是新的手藝活。
model 每幾個月就換一代,你永遠追不完,但你為自己工作流打造的 harness,那些 rules, skills, evols 是存在 version control 裡,會負利的資產。
model 越換越強,你的系統跟著水漲穿高,這是我認為工程師現在最值得投資時間的地方。
google 的這門課程拆成五天,這支影片講的是 day 1 的課程內容,而 day 2 講的是 agent 工具, mcp 跟 a2a, day 3 講 skills, ge 還有 context 優化。
day 4 講 security 跟 evaluation, day 5 講 spec driven 的 production level 開發。
